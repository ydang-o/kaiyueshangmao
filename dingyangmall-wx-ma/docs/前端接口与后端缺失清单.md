# 前端接口与后端缺失清单

本文档以 **README.md（后端接口文档）** 为基准，整理 **小程序端** 与 **Android/iOS 端** 应使用的全部接口，并列出 **后端需补充或可能缺失** 的接口与实现说明，便于查漏补缺。

**请求根地址**：各端配置的 basePath（如小程序 `config/env.js` 的 `basePath`），默认 `http://localhost:7500`。

---

## 一、端与路径约定（与后端 README 一致）

| 端 | 路径前缀 | 认证方式 | 说明 |
|----|----------|----------|------|
| **管理端** | 无统一前缀（如 `/system`、`/orderinfo`） | `Authorization: Bearer {token}` | 仅 PC 管理端，本清单不列 |
| **移动端（小程序 / Android / iOS）** | `/weixin/api/ma/*`、`/app/*` | 小程序：`app-id` + 登录后 `third-session`；Android/iOS：同逻辑用 Token | **三端共用同一套接口路径与协议** |
| **商家端** | `/api/mall/merchant/*`、`/login`、`/captchaImage` | `Authorization: Bearer {merchantToken}` | 商家扫码、核销、登录 |

**说明**：Android、iOS 与小程序对接**同一套**下表接口即可，仅请求头中登录态字段名可能不同（如 Token 名）。

## 四、后端缺失或需确认的接口/实现

以下为前端已使用、但后端 README 未单独列出或与管理端实现可能不一致的项，**需在后端代码中查漏补缺**。

### 4.1 明确需移动端专用路径（易漏）

| 序号 | 接口 | 说明 | 建议后端处理 |
|------|------|------|--------------|
| 1 | **GET** `/weixin/api/ma/notice/list` | 首页公告列表。管理端仅有 `/system/notice/list`，移动端需独立接口或网关转发。 | 在 **weixin/api/ma** 下提供公告列表（可复用 system 业务，返回给 C 端的数据结构），或网关将 `/weixin/api/ma/notice/list` 映射到现有公告服务。 |
| 2 | **GET** `/app/member/send-sms-code` | 移动端发送短信验证码。管理端为 **GET** `/common/sms/send`。 | 在 **/app/member** 下实现 `send-sms-code`（Query: phone），或统一由网关转发到 `/common/sms/send`，保证移动端只调 `/app/member/send-sms-code`。 |
| 3 | **GET** `/app/lottery/config`、**POST** `/app/lottery/draw`、**GET** `/app/lottery/record` | 抽奖。管理端为 `/mall/lottery/config`、`/mall/lottery/record/page`，且无 draw。 | 在 **/app/lottery** 下实现 config、draw、record（与 /mall/lottery 业务可复用，路径与权限区分）。 |

## 五、前端（小程序）实际调用与 README 对齐情况

- **utils/api.js** 中所有请求路径已与本文档 **第二节** 一致。
- **Android/iOS**：对接时使用上表相同 **方法 + 路径**，仅请求头中登录态字段按各端约定（如 Token 名）即可。
- 分页参数：前端使用 `current`、`size`，与 README 中「分页接口一般支持 current/size 或 pageNum/pageSize」一致，后端需至少支持其一并统一。

## 六、汇总：后端需补全/核对清单（按优先级）

1. **必须实现（移动端独有路径）**  
   - `/weixin/api/ma/notice/list`（公告列表）  
   - `/app/member/send-sms-code`（短信验证码）  
   - `/app/lottery/config`、`/app/lottery/draw`、`/app/lottery/record`（抽奖）

2. **必须支持（与前端约定一致）**  
   - `/weixin/api/ma/wxuser/login` 由 API 处理，不被静态资源拦截  
   - `/weixin/api/ma/useraddress` 的 POST 在带 `id` 时做**修改**  
   - `/weixin/api/ma/shoppingcart/del` 的 **POST** 删除（body 传 id/ids）

3. **建议核对**  
   - 订单：receive、logistics、countAll、unifiedOrder、refunds 的路径、方法、入参与返回  
   - 商家端：`/captchaImage`、`/login` 的访问与鉴权

按上述清单在后端逐项实现或核对后，前端（小程序 + Android/iOS）即可与后端接口完全对齐，便于联调与上线。
